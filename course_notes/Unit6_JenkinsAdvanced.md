UNIT VI: CI/CD USING JENKINS
WEEK 17: INTEGRATING DOCKER, MAVEN & JENKINS PIPELINES

Section A — RUNNING BUILDS INSIDE DOCKER CONTAINERS
A common CI/CD challenge is managing dependencies across different projects. If Project A requires Node 14 while Project B needs Node 18, installing both on the same Jenkins agent creates conflicts.
The solution is to use Docker as the build environment.
In a Declarative Pipeline, you can specify a Docker image as the agent. Jenkins will automatically pull the container, mount the Git workspace, run your `sh` commands inside the container, and dispose of it when done. This guarantees a clean, isolated, and reproducible build environment for every run.

Section B — MAVEN INTEGRATION WITH JENKINS
Maven can be deeply integrated into Jenkins workflows:
- Global Tool Configuration: Jenkins administrators can register specific Maven and JDK versions through the system settings.
- Executing Maven Builds: Use `sh 'mvn clean package'` to invoke Maven from pipeline steps.
- Test Reporting: The `junit` step parses XML test reports from `target/surefire-reports/*.xml` (generated by Maven Surefire) and displays visual trend charts on the Jenkins dashboard showing pass/fail history over time.

Section C — BUILD TRIGGERS & DEPLOYMENT STRATEGIES
What initiates a build?
- Webhooks (Push-based): GitHub sends an HTTP POST request to Jenkins the moment code is pushed. This is instantaneous and resource-efficient.
- Polling (pollSCM): Jenkins periodically checks Git for changes (e.g., every minute). This approach wastes compute resources.
Deployment steps typically involve shell scripts calling `kubectl apply` for Kubernetes clusters or Ansible playbooks for server provisioning.

Section D — SHARED PIPELINE LIBRARIES
When an organization manages 50 microservices, maintaining 50 nearly identical Jenkinsfiles violates the DRY (Don't Repeat Yourself) principle.
Jenkins Shared Libraries let you define reusable Groovy functions (e.g., `standardJavaPipeline()`), host them in a dedicated Git repository, and import them globally. Individual project Jenkinsfiles can then be reduced to a single line calling the shared library function.

Section E — BACKUP STRATEGY & PIPELINE BEST PRACTICES
- Jenkins Configuration as Code (JCasC): Define Jenkins system configuration in YAML files instead of manually configuring through the UI.
- Backup: The `$JENKINS_HOME` directory contains all critical data. Schedule regular backups.
- Keep Pipelines Simple: Avoid embedding hundreds of lines of complex shell logic in a Jenkinsfile. Instead, create a `deploy.sh` script in the repository and have Jenkins call it with `sh './deploy.sh'`. This lets developers test the exact deployment process locally.

### Lab Exercise 5: Running a Pipeline Inside a Docker Container
**Scenario:**
Create a Jenkinsfile where the build agent dynamically uses the `maven:3.8.4-openjdk-11` Docker image to compile the project, eliminating the need to install Maven on the Jenkins host machine.

**Code:**
```groovy
pipeline {
    agent {
        docker { 
            image 'maven:3.8.4-openjdk-11' 
            args '-v $HOME/.m2:/root/.m2' // Persist Maven cache on the host filesystem
        }
    }
    stages {
        stage('Compile') {
            steps {
                sh 'mvn --version'
                sh 'mvn clean package -DskipTests'
            }
        }
    }
}
```
*Note:* Jenkins handles image pulling automatically and wraps the `steps` block inside a `docker run` invocation transparently.

### Lab Exercise 6: Processing JUnit Test Reports
**Scenario:**
After executing `mvn test`, Maven produces XML test result files. Add a `post` block to your pipeline that leverages the Jenkins JUnit plugin to parse these XML reports and display them in the Jenkins build UI.

**Code:**
```groovy
pipeline {
    agent any
    stages {
        stage('Test') {
            steps {
                // Allow pipeline to continue to post block even if tests fail
                catchError(buildResult: 'SUCCESS', stageResult: 'FAILURE') {
                    sh 'mvn test'
                }
            }
        }
    }
    post {
        always {
            // Consume and display the XML test reports
            junit 'target/surefire-reports/*.xml'
        }
    }
}
```
*Note:* The `junit` step reads XML files and populates the "Test Result" section in the Jenkins dashboard. Using `catchError` ensures the pipeline reaches the `post` block even when individual tests fail.
